Technical Documentation

JetUse 技術詳細ドキュメント

2026年6月22日時点の再調査内容に基づき、JetUseの実装、設定、IaC、運用スクリプトを根拠として、OCI上の生成AI利用環境をカテゴリ別に整理したエンジニア向け資料です。

ドキュメントの位置づけ OCI Generative AI、Autonomous Database、Object Storage、API Gateway、Functions、IAM Identity Domain、OCI Speech、OCI Languageなどを組み合わせたOCI実装テンプレートの技術詳細です。

全体像

この章では、JetUseリポジトリを構成する主要パッケージ、IaC、運用補助、仕様メモの責務を整理します。実装確認では、画面・API・共有ロジック・Terraform・運用スクリプトを分けて追うと全体像をつかみやすくなります。

JetUse は、OCI 上でチャット、RAG、DBチャット、保存済みエージェント、音声議事録、リアルタイム文字起こし、音声チャット、動画フレーム分析、OCR、管理ダッシュボードを提供する生成AI利用環境です。フロントエンドは React SPA、バックエンドは Container Instance 上の FastAPI と OCI Functions ルーター、AI 機能は OCI Generative AI の OpenAI 互換 API、データ永続化は Autonomous Database 26ai と Object Storage を中心に構成されています。

画像生成・動画生成は実装対象外です。画像入力、動画フレーム分析、OCR は利用シナリオとして整理されています。

パス役割
packages/web/React 19 + Vite + Tailwind CSS v4 の SPA。HashRouter、OIDC PKCE、各ユースケース画面を実装
packages/api/FastAPI service、OCI Functions router、共通ドメインロジック、DB マイグレーション、API テスト
packages/shared/API と agent container で共有する SQL ガード、Web 取得、SSRF 対策
packages/agent-containers/OpenAI Agents SDK / Google ADK / LangGraph 用の Hosted Application コンテナ
infra/terraform/OCI リソース定義。共有 dev 環境と開発者別 app 環境、各モジュール
ops/SQL Search、Select AI、Hosted Agent、開発者別環境、ADB 起動などの運用補助
docs/アーキテクチャ、比較、ADR、セットアップ、検証、運用 Tips
specs/フェーズ別・機能別の仕様メモ
spikes/実機検証用のスパイクコード

フロントエンド

フロントエンドは packages/web の SPA です。React Router v7 の HashRouter を使い、API Gateway + Object Storage の静的配信でディープリンク時のサーバー 404 を避けています。App.tsx では各ページを React.lazy() で遅延ロードし、Shell は常時表示します。

主な画面と API は次の通りです。

画面主な API / 処理
chat.tsx/api/chat/stream, /api/conversations, /api/presets, /api/agent/*, /api/agents/*
rag.tsx/api/rag/files, /api/chat/stream。RAG backend は vector_store / select_ai / opensearch
dbchat.tsx/api/chat/nl2sql, /api/dbchat/schema, /api/dbchat/execute, /api/dbchat/chart, /api/db/datasets*
agentbuilder.tsx/api/agents*, /api/agent/tools, /api/agent/select-ai-tools, /api/agents/projects
minutes.tsx/api/minutes, /api/minutes/{id}/generate
realtime.tsx/api/stt/sessions*, /api/translate
voicechat.tsxSTT API、/api/chat/stream、/api/tts
video.tsxブラウザ側で動画フレームを画像化し、images として /api/chat/stream に送信
ocr.tsx/api/ocr/options, /api/ocr
builder.tsx / usecase.tsx/api/usecases*, /api/tools/extract-url, /api/chat/stream
admin.tsx/api/admin/usage

認証は VITE_AUTH_REQUIRED=true の場合に oidc-client-ts を使います。silent renew は使わず、期限前または 401 時にトップレベルの signinRedirect() を行う設計です。認証無効時は dev-user として動作します。

SSE は EventSource ではなく、fetch() の ReadableStream を共通パーサ packages/web/src/lib/sse.ts で読みます。POST ボディ、Bearer ヘッダ、AbortController、401 再認証の扱いを画面ごとに制御するためです。

バックエンド API 構成

FastAPI serviceを主経路、OCI Functions routerを一部APIの補助経路として使う構成です。SSE、アップロード、長時間処理はContainer Instance側、presets・dbchat・ttsの一部はFunctions側という責務で確認します。

バックエンドは 2 系統です。

  1. FastAPI service
  • packages/api/service/main.py
  • Container Instance 上で uvicorn service.main:app として動作
  • SSE、長時間処理、ファイルアップロード、リアルタイム STT のプロセス内状態を担当
  1. OCI Functions router
  • packages/api/fn/router/func.py
  • presets、一部 dbchat、tts を担当
  • 共通ドメインロジックを共有して二重実装を避ける

FastAPI は service/routes 配下へ分割されています。service/main.py は middleware、DB 例外ハンドラ、/healthz、router include の薄い組み立て層です。SSE ヘッダと keepalive フレームは service/sse.py、リクエスト DTO は service/schemas.py、保存済みエージェントのディスパッチは service/agent_dispatch.py に分かれています。

API Gateway のルートは次の設計です。

ルート行き先timeout備考
/api/chat/{p*}Container Instance300秒チャット、RAG、NL2SQL など SSE を含む
/api/ocrContainer Instance300秒多ページ OCR の同期処理用に完全一致ルート
/api/{p*}Container Instance60秒CI 側の一般 API catch-all
/api/presets, /api/presets/{p*}OCI FunctionsFunctions 側fn_router_segments で定義
/api/dbchat, /api/dbchat/{p*}OCI FunctionsFunctions 側schema / execute / chart を router が担当
/api/tts, /api/tts/{p*}OCI FunctionsFunctions 側MP3 応答
/, /{object*}Object Storage PAR-SPA 静的配信

なお、開発者別 app 環境では functions_routes={} として全 /api を本人の Container Instance に流す構成が infra/terraform/environments/app/main.tf にあります。

OCI リソース定義

Terraform は共有 dev 環境と開発者別 app 環境に分かれています。

共有 dev 環境 infra/terraform/environments/dev/main.tf は、VCN、Object Storage、ADB、OCIR、Container Instance、OpenSearch、Functions、Logging、API Gateway、Identity Domain、IAM を組み立てます。OpenSearch は enable_opensearch=false が既定で、常設課金を伴うため明示的に有効化する設計です。

開発者別 app 環境 infra/terraform/environments/app/main.tf は、共有 state を terraform_remote_state で参照し、開発者ごとの Container Instance、API Gateway、SPA バケットだけを作ります。ADB は共有し、ADB_USER / ADB_QUERY_USER で専用スキーマへ切り替えます。

主な Terraform モジュールは次の通りです。

認証・認可

認証有効時は、SPA が IAM Identity Domain に OIDC Authorization Code + PKCE でログインし、API に Bearer token を送ります。API は auth.py の verify_token() で JWKS から署名鍵を取得し、RS256 JWT を検証します。AUTH_REQUIRED=false の場合は AuthContext(subject="dev-user") を返します。

アプリ内データは owner_sub を中心に分離します。会話、RAG、MCP サーバー、エージェント、議事録、ユースケース、プリセットは SQL の WHERE owner_sub = :o または visibility='public' 条件で制御されています。管理画面は ADMIN_USERS に JWT sub または email が含まれるかで判定します。グループ claim やアプリロールによる細かい RBAC は、このリポジトリでは確認できません。

MCP サーバー登録は認証なし URL のみ実装されています。auth_token 付き登録は Vault 書き込み権限が必要なため、API は 501 を返します。

LLM 呼び出しとモデル管理

LLM クライアントは genai.py に集約されています。AUTH_MODE=resource_principal のときは OCI Resource Principal、そうでないときは OCI config の User Principal で IAM 署名します。OpenAI SDK には api_key="OCI" のダミー値と署名済み httpx.Client を渡します。

OpenAI 互換 API のホストは 2 系統です。

用途URL実装
推論、Responses、Chat Completions、Files、Conversations、File Searchhttps://inference.generativeai.{region}.oci.oraclecloud.com/openai/v1make_inference_client()
Vector Store 本体 CRUDhttps://generativeai.{region}.oci.oraclecloud.com/20231130/openai/v1make_cp_client()

モデルは models.py の静的レジストリで管理します。

keyOCI model idAPI family備考
gpt-oss-120bopenai.gpt-oss-120bResponses標準。reasoning effort 対応
llama-3.3-70bmeta.llama-3.3-70b-instructChat Completions軽量・高速
gemini-2.5-progoogle.gemini-2.5-proChat CompletionsVision 対応、複数画像対応、min_max_tokens=2048
gemini-2.5-flashgoogle.gemini-2.5-flashChat CompletionsVision 対応、複数画像対応
llama-3.2-90b-visionmeta.llama-3.2-90b-vision-instructChat CompletionsVision 対応、複数画像は不可

chat.py は Responses と Chat Completions を {"delta"}, {"usage"}, {"citations"}, {"error"} のイベント列へ正規化します。Responses 系では top_p、max_output_tokens、reasoning.effort、file_search tool を扱います。Chat Completions 系では top_p、max_tokens、stream_options.include_usage を扱い、Gemini 系の max_tokens は実用下限でクランプします。

埋め込みは OpenAI 互換 /embeddings ではなく、OCI ネイティブ SDK の GenerativeAiInferenceClient.embed_text() を使います。モデルは cohere.embed-multilingual-v3.0、1024 次元です。

ストリーミング応答

SSE は FastAPI の StreamingResponse で実装されています。SSE_HEADERS は Cache-Control: no-cache と X-Accel-Buffering: no を付与し、keepalive はコメント行ではなく data: {"ka": 1} フレームです。間隔は 15 秒です。

中心ルート service/routes/chat.py では、上流 LLM の同期ストリームを producer thread で読み、asyncio.Queue 経由で async generator に流します。クライアント切断時は threading.Event を立て、上流 stream を close() します。途中まで出た assistant 応答は、会話 ID がある場合に ADB へ保存されます。

Select AI RAG、OpenSearch RAG、Hosted Agent、Select AI Agent は内部的に非ストリーミング実行ですが、API は keepalive を送りながら完了を待ち、結果を単発の delta として SSE に載せます。UI からは同じ読み取り経路で扱えます。

会話履歴・データ永続化

永続化は Autonomous Database 26ai です。DB 接続は mTLS ウォレットを使い、ADB_WALLET_DIR があればローカルディレクトリ、なければ Object Storage の非公開バケットから ADB_WALLET_OBJECT を取得して /tmp/adb_wallet に展開します。DB 停止時にハングしないよう、接続確立、プール取得、SQL 往復にタイムアウトが設定されています。

マイグレーションは migrate.py が migrations/*.sql を辞書順に適用し、schema_migrations に記録します。主なテーブルは次の通りです。

テーブル保存内容
conversations会話メタデータ、所有者、モデル、OCI Conversation ID
messages会話内メッセージ。conversation_id への FK と cascade delete
usage_log会話単位の token usage
prompt_presetsユーザー別 system prompt プリセット
usecasesユースケース定義 JSON と公開範囲
rag_storesユーザー別 Vector Store ID
rag_filesRAG ファイル、OCI file id、状態、サイズ、エラー
mcp_serversMCP サーバー URL と Vault secret OCID
agentsエージェント定義、tools、Project OCID、framework
minutes_jobs音声ファイル、Speech job、transcript、状態
audit_log機能、モデル、トークン、状態、メタ情報

短期メモリは Responses 系モデルで OCI Conversations を使います。会話 ID に紐づく OCI Conversation が未作成なら、metadata.short_term_memory_optimization=true と memory_subject_id=user.subject を含めて作成し、ADB に ID を保存します。Project 側の long-term memory 設定そのものは Terraform では確認できないため、長期メモリの有効化状態はこのリポジトリだけでは確認できません。

RAG / ナレッジベース / ファイルアップロード

RAGはVector Store、Select AI、OpenSearchの3バックエンドを実装しています。どのバックエンドで回答させるか、ファイルの登録状態をどこで見るか、反映遅延が正常範囲かを切り分ける章です。

アプリ実装上の RAG backend は 3 つです。

backend実装特徴
vector_storeOCI OpenAI 互換 Files + Vector Store + Responses file_search既定。ストリーミングと citations に自然に統合
select_aiADB DBMS_CLOUD_AI の Select AI with RAGObject Storage バケット直結。同期 SQL 関数で回答
opensearchOCI Search with OpenSearch + OCI GenAI embedding + Chat Completionsk-NN 検索を自前実装。常設クラスタ費がある

RAG ファイルアップロードは service/routes/rag.py で受け、拡張子は .pdf, .txt, .md のみ、上限は 20MB です。.docx は明示的に 422 で拒否されます。アップロード時は Vector Store 登録、Object Storage への原本バックアップ、OpenSearch が有効な場合の best-effort ingest が行われます。Select AI RAG は Object Storage の rag/{owner}/ を参照し、profile/index を遅延作成します。

GET /api/rag/files は各ファイルに backends 状態を付与します。Vector Store は Files API の処理状態、Select AI は {INDEX}$VECTAB の attributes.object_name、OpenSearch は index の file_id 集約で判定します。Select AI は refresh_rate 間隔のバケット同期で反映が遅れるため、UI では pending が長く続くことがあります。

ナレッジベース系の拡張は設計候補として整理されていますが、現在のアプリ実装では上記3バックエンドを主経路として扱います。

実装確認メモAgents Knowledge Baseは比較対象・設計候補として文書にありますが、このサイトで案内するRAG主経路はVector Store、Select AI、OpenSearchです。運用確認ではファイル状態APIと各バックエンドの反映状態を優先して見ます。

DBチャット / NL2SQL / CSVデータセット

DB Chat は自然言語から SQL を生成し、ユーザーが確認・編集してから読取専用ユーザーで実行する構成です。SQL 生成 backend は 2 つです。

backend実装備考
sql_searchGenerative AI Semantic Store の generateSqlFromNlSH サンプル向け。SQL Search セットアップが必要
select_aiADB DBMS_CLOUD_AI.GENERATE(action=>'showsql')SH またはユーザー CSV データセット用 profile

SQL 実行は 読取専用ユーザーで行います。さらに 共有SQLガード が SELECT / WITH のみ許可し、更新系キーワード、複数ステートメント、コメントを拒否します。結果は最大 200 行、セルは 300 文字に制限されます。

CSV データセットは POST /api/db/datasets で取り込みます。CSV は アプリスキーマに表として作成され、読取専用ユーザーに SELECT を grant し、本人専用 Select AI profile を再構築します。AI サンプルデータ生成は gemini-2.5-flash で CSV を生成し、同じ create_dataset() に流します。

エージェント

保存済みエージェントはHosted Application系3種とADB Select AI Agentの計4系統で整理します。フレームワークのcanonical値、旧値の正規化、MCP対応範囲を確認する章です。

保存済みエージェントは 2 系統 4 種です。

framework実行場所実装ツール
openai_agentsOCI Hosted ApplicationOpenAI Agents SDK + OpenAIChatCompletionsModelweb_search, web_fetch, get_current_time, rag_search, query_database
adkOCI Hosted ApplicationGoogle ADK + カスタム BaseLlm同上
langgraphOCI Hosted ApplicationLangGraph create_react_agent同上

AgentDefinition.framework の canonical 値は openai_agents, adk, langgraph, select_ai です。旧値 native, agents_sdk, hosted はマイグレーション 012_normalize_framework.sql で openai_agents に正規化されます。読取時には hosted_agent.normalize_sdk() に後方互換マッピングが残っています。

保存済みエージェント実行は service/agent_dispatch.py で、select_ai は ADB の Select AI Agent、その他は Hosted Application へディスパッチします。Hosted 3 種は非ストリーミング invoke なので、API は keepalive を送りながら待ち、tool_trace と本文を SSE の delta として流します。

アドホックなツール付きチャットは chat.py の stream_agent() が現役です。これは保存済みエージェントの主経路ではありませんが、チャット画面で選んだ tool / MCP を Responses API の function tool として使う用途で残っています。

MCP サーバーは登録 API と UI が存在しますが、保存済み hosted agent containers は MCP 未対応として validator で拒否します。アドホック agent mode では MCP server ids を解決して Responses API に渡す実装があります。

音声・翻訳・動画・OCR

音声議事録、リアルタイムSTT、音声チャット、翻訳、動画フレーム分析、OCRの実装経路をまとめます。同期処理かSSEか、OCIサービス失敗時にフォールバックがあるか、対応ファイル形式と上限を確認します。

議事録

議事録は POST /api/minutes で音声を Object Storage に保存し、OCI Speech のバッチ文字起こしジョブを作成します。対応拡張子は .mp3, .wav, .m4a, .ogg, .webm、上限は 100MB です。Speech の出力 JSON は一覧・詳細取得時にポーリング同期され、トークンを話者ごとの発話へ結合します。議事録生成は transcript から LLM で Markdown を SSE 生成します。

リアルタイム STT / 音声チャット

リアルタイム STT は、クライアントから音声 PCM チャンクを POST /api/stt/sessions/{sid}/audio に送り、結果は GET /api/stt/sessions/{sid}/events の SSE で受けます。API と OCI Speech の間は oci-ai-speech-realtime の WebSocket です。セッションはプロセス内保持なので、Container Instance 1台構成が前提です。

音声チャットは、この STT 基盤で確定テキストを作り、通常の /api/chat/stream と /api/tts を組み合わせる半二重の構成です。フルデュプレックス音声モデルの実装は確認できません。

翻訳

翻訳は llm と oci_language の 2 backend です。llm は llama-3.3-70b、oci_language は OCI Language batch translation を使います。OCI Language が IAM 未付与などで失敗した場合は LLM へフォールバックします。

動画・画像入力

動画分析はブラウザ側でフレームを data URI 画像として抽出し、/api/chat/stream の images に渡します。API は vision 対応モデルだけを許可し、agent/rag との併用を拒否します。llama-3.2-90b-vision は複数画像不可、Gemini 系は複数画像可としてモデルレジストリに表現されています。

OCR

OCR は 2 engine です。

engine実装特徴
document_understandingOCI Document Understanding analyze_document 同期 API日本語テキスト抽出。表抽出は英語のみ。5ページ超 PDF は 5ページ単位に分割して並列 OCR
vlmVision LLM にページ画像を渡して JSON / text を生成日本語表に対応しやすいが、ページごとに LLM 呼び出し。厳密 OCR ではない

OCR の upload 対応形式は PNG/JPEG/TIFF/PDF、アップロード全体の上限は MAX_TOTAL_BYTES=60_000_000、総ページ上限は 100 です。

ADB、Compute、Functions、Object Storage、OCI Generative AIの利用量を確認します。検証後に放置しない前提で進めます。

権限

検証用コンパートメントを使い、Stack作成者、アプリ利用者、管理者を必要最小限にします。

保存データ

会話履歴、RAGファイル、議事録、エージェント定義、ログが保存される前提でサンプルデータを使います。

共有前確認

画面を見せる前に、ログインユーザー、公開範囲、サンプルデータ、出力内容を確認します。

Cleanup

試した後に消すもの

まずStackをDestroyし、その後にデータやログの残りを確認します。検証用コンパートメントを使うと削除範囲が分かりやすくなります。

Resource Manager Stack

試用が終わったら、作成したStackをDestroyします。失敗した場合はエラー内容を確認し、残ったリソースを個別に確認します。

Object Storage

RAGファイル、音声ファイル、静的配信バケット、生成データが残っていないか確認します。

ADBとログ

会話履歴、議事録、エージェント定義、監査ログ、サービスログが不要であれば削除対象に含めます。

本番化は別設計

監査、バックアップ、保存期間、可用性、アクセス制御は、本番要件に合わせて別途設計します。

FAQ

よくある質問

JetUseは誰向けのテンプレートですか？

OCI上で生成AIの利用環境を短時間で試したいチーム向けです。チャット、RAG、DBチャット、議事録、エージェントなどを動かしながら、自社ユースケースに合うかを確認するための入口として使います。

最初はどの構成を選べばよいですか？

迷う場合は標準構成を選びます。チャット、RAG、DBチャット、議事録まで確認でき、初回検証に向いています。

Deploy to Oracle Cloudボタンはどこにありますか？

GitHubリポジトリのREADMEに配置します。このサイトでは、ボタンを押した後に何を確認するかを案内します。GitHubリポジトリを開く

利用すると費用は発生しますか？

はい。OCIリソースを作成し、生成AI、Autonomous Database、Object Storage、API Gateway、Container Instanceなどを利用するため、検証中も利用量に応じた費用が発生する可能性があります。検証用コンパートメントを分け、不要になったら削除します。

どのOCIリソースが作成されますか？

構成により異なりますが、Resource Manager Stackから、アプリ実行基盤、API Gateway、Object Storage、Autonomous Database、OCI Generative AI連携、認証関連リソースなどを作成します。詳細は技術ドキュメントのOCIリソース定義を確認します。

ログインできない場合は何を確認しますか？

まずURLが正しいか確認します。次に、Identity Domainのユーザー、クライアント設定、リダイレクトURIを確認します。詳しい確認は技術ドキュメントの認証項目を参照します。

チャットの回答が返ってこない場合は何を見ますか？

モデルを利用できるリージョンか、API Gatewayとバックエンドが起動しているか、OCI Generative AIの権限があるかを確認します。

RAGでファイルをアップロードした後、すぐ質問できますか？

ファイル処理が完了してから質問します。処理中や失敗の表示がある場合は、ファイル形式、サイズ、Object Storageの設定を確認します。

社内データや機密情報を入れてもよいですか？

検証段階では、社内ルールに従い、投入してよいデータだけを使います。機密情報、個人情報、顧客データを扱う場合は、保存先、アクセス権限、ログ、削除手順、利用するモデルやリージョンを事前に確認します。

本番環境としてそのまま使えますか？

まずは検証・PoC向けの実装テンプレートとして扱います。本番利用する場合は、セキュリティ、監査、運用監視、可用性、バックアップ、コスト管理、社内承認プロセスを確認したうえで、必要な設計変更を行います。

自由に使ってよいですか？責任範囲はどうなりますか？

テンプレートは検証や自社ユースケース検討の参考として利用・改変できます。ただし、利用結果、本番適用、費用発生、データの取り扱い、設定変更による影響については利用者側で確認・判断してください。本サイトの内容は利用者環境での動作や結果を保証するものではありません。

試用後は何を削除しますか？

Resource Manager StackをDestroyし、Object Storage、ADB、ログ、生成したデータが残っていないか確認します。検証用コンパートメントを使うと後片付けが楽になります。

活用支援はどこに相談できますか？

検証シナリオの整理、OCI上での実装・展開、生成AI機能の使い分けなどの支援が必要な場合は、日本オラクル クラウド事業本部 AI Solution Accelerationチームまでご相談ください。
